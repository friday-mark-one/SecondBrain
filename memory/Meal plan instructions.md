# Vault Operating Manual (for Claude Code)

This vault runs a deterministic meal-planning + grocery system, contained in the
`memory/` folder. Follow these procedures exactly. Prefer exact paths over
searching. When a request is ambiguous, STOP and ask ‚Äî list the candidate files;
never guess or bulk-edit. All paths below are relative to the vault root.

## The notes ARE the source of truth
There is no separate database or import file. The engine reads the notes
directly ‚Äî `Items/` frontmatter, `Recipes/` ingredients, `Buy List.md`,
`Meal Plan/Current.md`. So to add or change food data, **edit the notes**
(via the QuickAdd commands, or by editing the note file). Whatever is in the
notes is what the grocery list reflects.

## Layout
- `memory/Items/<Name>.md` ‚Äî one note per ingredient/staple/buy list. Frontmatter:
  `type: item`, `store: <Costco|Fred Meyer|Indian Store|Trader Joes|Bath & Body works>`,
  optional `category`. The note name IS the `[[wikilink]]` recipes use.
  - Optional expiry fields (opt-in, perishables only): `shelf_life_days` (int ‚Äî
    its presence is what makes the item tracked), `heads_up_days` (int, days
    before expiry to start warning; absent ‚áí 0 = warn on expiry day only), and
    `expires` (`YYYY-MM-DD`, **engine-stamped** at Archive & Reset ‚Äî never
    hand-edit). Items with no `shelf_life_days` are not tracked.
- `memory/Recipes/<Dish>.md` ‚Äî `type: recipe` + `meal`/`cuisine`/`protein_heavy`/
  `fodmap_friendly` + a `## Ingredients` section of `- [[Item]] | amount` lines +
  `## Directions`.
- `memory/Meal Plan/Current.md` ‚Äî the single active plan (day ‚Üí slot headings).
- `memory/Grocery List/Current.md` ‚Äî generated by the Generate command; only ever
  tick boxes here, don't hand-author it.
- `memory/Buy List.md` ‚Äî non-recipe items, in two sections (lines MUST contain a
  `[[Item]]` link or they're ignored):
  - `## One-off` ‚Äî throwaway items; ALL added to the list; removed once bought.
  - `## Regulars` ‚Äî a persistent menu of `- [ ] [[Item]]` checkboxes; only the
    **ticked** (`- [x]`) ones are added that week. Items always persist; ticks
    reset on Archive & Reset. Bought regulars are NOT removed or archived.
  - (If neither heading exists, the whole file is treated as One-off.)
- `memory/Cookbook.md` ‚Äî DataviewJS browse + ‚ûï buttons.
- `memory/Expiring Soon.md` ‚Äî DataviewJS view of tracked items inside their
  heads-up window, soonest first (read-only; just reads `expires`).
- `memory/Cooking References.md` ‚Äî rice/upma ratios, Instapot timings (NOT recipes).
- `memory/_config.md` ‚Äî `store_order`, `meal_slots`, `default_plan_days`,
  `grocery_detail` (`full` | `off`).

## QuickAdd commands (the only way to run the engine)
- **Generate Grocery List** ¬∑ **Toggle Grocery Detail** (Full‚ÜîOff) ¬∑
  **Add Dish To Plan** (Cookbook ‚ûï) ¬∑ **New Weekly Plan** ¬∑ **Archive And Reset** ¬∑
  **New Item** ¬∑ **New Recipe** (these two create a note from `_templates/`).
- Never reproduce these by hand-editing files.

## Procedures
- **Add an item:** QuickAdd **New Item** (or create `Items/<Title Case Name>.md`
  from `_templates/Item.md`); set its `store`.
- **Change an item's store:** edit only that one `Items/<Name>.md` `store` field.
- **Add / edit a recipe:** QuickAdd **New Recipe** (or edit the
  `Recipes/<Dish>.md` note directly). Ingredients are `- [[Item]] | amount`
  lines; create any missing item note first so the link resolves.
- **Buy a one-off item this week:** append `- [[Item]] | amount` under
  `## One-off` in `Buy List.md`.
- **Add a recurring item to the menu:** append `- [ ] [[Item]] | amount` under
  `## Regulars` in `Buy List.md` (unticked). To include it a given week, tick it.
- For either, if the item note is missing, ask before creating `Items/<Item>.md`.
- **Track an item's expiry:** add `shelf_life_days` (and optionally
  `heads_up_days`) to that perishable's `Items/<Name>.md`. Don't set `expires` by
  hand ‚Äî it's stamped when you check the item off at Archive & Reset.
- **Bulk edit items/recipes:** transform the actual notes in place (read
  `Items/`/`Recipes/`, edit, write back). NEVER regenerate notes from a separate
  source file ‚Äî that path caused drift and has been removed.
- **Which recipes use an item:** `grep -rl "\[\[<Item>\]\]" memory/Recipes/`.

## The engine (only regenerate-able part)
- `memory/_scripts/mealplan.js` is the engine source (pure helpers + adapters).
- `memory/_scripts/commands/*.js` are GENERATED from it ‚Äî **do not hand-edit
  them.** To change engine behavior: edit `mealplan.js`, update/add tests, run
  `node --test` (must stay green), then `node build-commands.js` to refresh
  `commands/`. The QuickAdd macros run the `commands/*.js` files.
- Tests + fixtures live in `_scripts/test/`.

## Naming & ingredient conventions
- Ingredient/staple line: `- [[Item]] | amount` (amount optional, free-form).
- **Skip salt & water** as ingredients. Keep named oils.
- Generic/unspecified cooking oil ‚Üí **`[[Groundnut oil]]`** (the default).
- Whole pepper ‚Üí **`[[Peppercorns]]`**; ground ‚Üí **`[[Pepper powder]]`**.
- Fenugreek **seeds** ‚Üí **`[[Fenugreek seeds]]`**; dried fenugreek **leaves** ‚Üí
  **`[[Kasuri methi]]`** (distinct items).
- Garlic is **`[[Garlic]]`** (not "Peeled Garlic"); roasted gram is
  **`[[Roasted gram]]`** (not "Odachakadalai").
- Keep one spelling per item; the note name is the exact link recipes use.

## Engine behaviors (for reference, don't reimplement)
- Grocery list de-dupes ingredients, groups by `store_order`, shows a per-dish
  breakdown. A dish planned N times collapses to `Dish (amount) √óN`.
- `grocery_detail: off` hides the breakdown (just `- [ ] Item`); Toggle flips it.
- Regenerating the list preserves existing checkmarks.
- Missing item note / blank store / unknown dish surface in ‚ö†Ô∏è sections,
  never silently dropped.
- Archive & Reset stamps `expires = today + shelf_life_days` on every **checked**
  grocery-list item that has a `shelf_life_days`. Unchecked/untracked items are
  left alone ‚Äî so re-buying resets expiry, not re-buying keeps it.

## Expiry notifier (mini-only, outside the engine)
- `memory/_scripts/notify-expiring.js` is a Node cron script that runs **only on
  the Mac mini** (not mobile), so it may use `require`/`fs`/`fetch`. It reuses the
  engine's tested `expiringSoon` helper and POSTs a Telegram message for items in
  their heads-up window. It is NOT a QuickAdd command and is NOT regenerated.
- Telegram credentials live OUTSIDE the vault at `~/.openclaw/telegram.json`
  (`{ "bot_token", "chat_id" }`) so the secret never syncs to the phone. Setup
  steps are in the script's header comment.

## Ambiguity rule
"Remove onion" is ambiguous (delete the item note? remove from a recipe? from
staples?). List the matches and ask which one. Same for "change pepper" (powder
vs peppercorns), "fenugreek" (seeds vs kasuri methi), etc.
